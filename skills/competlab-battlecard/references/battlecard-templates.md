> Reference for the competlab-battlecard skill. Format variations and sales enablement heuristics.

## Battlecard Format Options

### Standard (1-page comparison)
Best for: individual competitor matchups, sales team distribution
The default format in the skill's output structure.

### Multi-Competitor Matrix
Best for: when the user faces 3-4 competitors simultaneously in deals

```
# Competitive Matrix: [Your Brand] in the [Category] Market

| Dimension | [Your Brand] | [Comp A] | [Comp B] | [Comp C] |
|-----------|-------------|----------|----------|----------|
| Price (starting) | | | | |
| Free trial | | | | |
| API access | | | | |
| [Key feature 1] | | | | |
| [Key feature 2] | | | | |
| Security grade | | | | |
| AI Visibility | | | | |

## When You're Against [Comp A]: [1-2 talking points]
## When You're Against [Comp B]: [1-2 talking points]
## When You're Against [Comp C]: [1-2 talking points]
```

### Persona-Specific Card
Best for: when the user sells to different buyer types

Adjust emphasis:
- **Technical buyer (CTO/dev):** Lead with tech stack, API, security, integrations
- **Business buyer (VP/PMM):** Lead with ROI, ease of use, time-to-value, customer logos
- **Executive buyer (CEO/founder):** Lead with market position, AI visibility, competitive moat

## Objection Handling Framework

Structure every objection response using the AER pattern:

1. **Acknowledge** — validate the concern (never dismiss)
2. **Evidence** — cite a specific fact or data point
3. **Redirect** — pivot to your strength

Example:
> **Objection:** "They have more integrations than you."
> **Response:** "That's true, they have a larger integration marketplace. [Acknowledge] What we've found is that most teams use 3-5 integrations, and ours cover the most common workflows — Slack, Jira, GitHub, and Salesforce are all native. [Evidence] The question is whether you need 200 integrations or whether you need the right 20 to work reliably. [Redirect]"

## "Killer Facts" Selection Criteria

The best killer facts for battlecards are:
1. **Verifiable** — the prospect can check this themselves
2. **Surprising** — something they wouldn't have noticed in a demo
3. **Relevant** — directly affects their use case or buying criteria
4. **Memorable** — simple enough to stick after the call ends

Bad: "We're a better product." (Vague, unverifiable)
Good: "Their security headers score an F on Mozilla Observatory. Ours score an A. You can check both right now." (Specific, verifiable, surprising)

## Landmine Questions That Work

Effective landmine questions share these traits:
- **Expose a real gap**, not a trivial one
- **The prospect can verify independently** (they'll lose trust if you're wrong)
- **Sound helpful, not attacking** ("Have you checked..." not "Did you know they don't...")

Categories of effective landmines:
| Category | Example |
|----------|---------|
| Missing feature | "Ask them to show you [feature] in their demo. It's important for [reason]." |
| Hidden cost | "Ask about their pricing for [overage/add-on]. It's not on their pricing page." |
| Technical limitation | "Ask how they handle [edge case]. This matters when you scale to [scenario]." |
| Support/SLA | "Ask about their response time guarantees. Check their status page history." |
| Data ownership | "Ask what happens to your data if you leave. Check their export options." |
